# 2.2. Módulo Notação UML – Modelagem Dinâmica


## Como este diagrama atende ao módulo

Este artefato representa o comportamento dinâmico do sistema no cenário "Iniciar Partida", evidenciando a troca de mensagens entre ator e componentes internos. O diagrama cobre fluxo principal e fluxo alternativo de erro, permitindo analisar responsabilidades, sequência temporal e possíveis falhas de execução.

## Descrição do Cenário Modelado

### Cenário
Iniciar partida a partir do menu principal.

### Participantes
- Jogador (ator externo)
- UI
- MenuController
- GameSession

### Fluxo Principal (Sucesso)
1. Jogador executa abrirJogo().
2. UI solicita carregarMenu() ao MenuController.
3. MenuController retorna menuCarregado.
4. Jogador seleciona selecionarOpcao("Iniciar").
5. UI solicita iniciarPartida() ao MenuController.
6. MenuController solicita create() para GameSession.
7. GameSession retorna sessaoCriada.
8. UI apresenta partidaIniciada ao Jogador.

### Fluxo Alternativo (Erro)
1. Durante iniciarPartida(), ocorre falha na criação da sessão.
2. Sistema retorna erroAoIniciar().
3. UI informa falha ao Jogador.

## Justificativas e Análise Crítica

A escolha do diagrama de sequência foi adequada porque o foco do cenário está na ordem das interações e nas responsabilidades entre componentes. O uso do bloco alternativo (alt) fortalece a modelagem ao demonstrar tratamento de exceção, não apenas o caminho ideal.

Pontos fortes:
- clareza do fluxo entre camadas (UI, controle e sessão);
- separação de responsabilidades;
- inclusão explícita de cenário de erro.

Pontos de melhoria:
- detalhar mensagens de retorno com dados de domínio (ex.: código de erro);
- explicitar pré-condições do caso (menu carregado, recursos disponíveis);
- futuramente complementar com diagrama de atividades para visão macro do processo.
